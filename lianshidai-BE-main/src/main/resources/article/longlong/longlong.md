## 四、模型总结

| 模型名称           | 优点                                                         | 缺点                                                         | 适用场合                                                     |
| :----------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| 瀑布模型           | - 线性模型<br>- 阶段间具有顺序性和依赖性<br>- 推迟实现的观点 | - 增加工作量<br>- 开发风险大<br>- 早期错误发现晚<br>- 不适应需求变化 | 系统需求明确且稳定、技术成熟、工程管理较严格的场合，如军工、航天、医疗。 |
| 原型模型           | - 减少需求不明确带来的风险                                   | - 构造原型采用的技术和工具不一定主流<br>- 快速建立起来的系统加上连续的修改可能导致原型质量低下<br>- 设计者在质量和原型中进行折中<br>- 客户意识不到一些质量问题 | 适用于需求不明确，需要通过原型来澄清需求的项目。             |
| 增量模型           | - 不需要提供完整的需求，只要有一个增量出现，开发就可以进行<br>- 软件能够更早投入市场<br>- 在项目的初始阶段不需要投入太多的人力资源<br>- 产品逐步交付，软件开发能够较好地适应需求的变化<br>- 开放式体系结构，便于维护<br>- 能够看到软件中间产品，提出改进意见，减少返工，降低开发风险 | - 每个增量必须提供一些系统功能，这使得开发者很难根据客户需求给出大小适合的增量<br>- 软件必须具备开放式体系结构（困难）<br>- 易退化成边做边改的方式，使软件过程控制失去整体性 | 适用于需求可能发生变化、具有较大风险、或者希望尽早进入市场的项目。 |
| 螺旋模型           | - 原型可看作可执行的需求规格说明，易于为用户和开发人员共同理解，还可作为继续开发的基础，并为用户参与所有关键决策提供了方便<br>- 螺旋模型为项目管理人员及时调整管理决策提供了方便，进而可降低开发风险 | - 如果每次迭代的效率不高，致使迭代次数过多，将会增加成本并推迟交付时间<br>- 使用该模型需要有相当丰富的风险评估经验和专门知识，要求开发队伍水平较高，否则会带来更大风险 | 适用于需求不明确或者需求可能发生变化的大型复杂的软件系统。   |
| 喷泉模型           | - 各个阶段没有明显的界限，开发人员可以同步进行开发，可以提高软件项目开发效率，节省开发时间，适应于面向对象的软件开发过程 | - 由于喷泉模型在各个开发阶段是重叠的，在开发过程中需要大量的开发人员，因此不利于项目的管理<br>- 喷泉模型要求严格管理文档，使得审核的难度加大，尤其是面对可能随时加入的各种信息、需求与资料的情况 | 适用于面向对象开发。                                         |
| 基于构件的开发模型 | - 软件复用思想<br />- 降低开发成本和风险，加快开发进度，提高软件质量 | - 模型复杂<br />- 商业构件不能修改，会导致修改需求，进而导致系统不能完全符合客户需求<br />- 无法完全控制所开发系统的演化<br />- 项目划分的好坏直接影响项目结果的好坏 | 适用于系统之间有共性的情况                                   |
| 敏捷开发           | - 快速响应变化和不确定性<br />- 可持续开发速度<br />- 适应商业竞争环境下的有限资源和有限时间 | - 测试驱动开发可能导致通过测试但非用户期望<br />- 重构而不降低质量困难 | 适用于需求模糊且经常改变的场合，适合商业竞争环境下的项目     |

- **关联关系**：体现的是两个类、或者类与接口之间语义级别的一种强依赖关系，例：关联类B以类属性的形式出现在关联类A中
- **泛化关系**：子类继承类或类实现接口
- **依赖关系**：一个类A使用到了另一个类B，而这种使用关系是具有偶然性，例：类B作为参数被类A在某个method方法中使用
- **组合关系**：体现整体与部分间的关系，整体的生命周期结束也就意味着部分的生命周期结束，例：人和四肢
- **聚合关系**：整体与部分之间是可分离的，他们可以具有各自的生命周期，部分可以属于多个整体对象，也可以为多个整体对象共享，例：汽车和轮子

语句覆盖
![image-20241209191430175](http://localhost:8080/article/material/longlong/longlong-images/image-20241209191430175.png)

- **语句覆盖**：就是设计若干个测试用例，运行被测程序，使得每一可执行语句至少执行一次。
- **分支覆盖（判定覆盖）**：就是设计若干个测试用例，运行被测程序，使得程序中每个判断的取真分支和取假分支至少经历一次。分支覆盖又称为判定覆盖。
- **条件覆盖**：设计若干个测试用例，运行被测程序，使得程序中每个判断的每个条件的可能取值至少执行一次。
- **条件组合覆盖**：设计足够的测试用例，运行被测程序，使得每个判断的所有可能的条件取值组合至少执行一次。
